# MyTaskKing — Prisma migrations

The schema in `schema.prisma` is the source of truth. Migrations live in `prisma/migrations/` and are generated by `prisma migrate dev`.

## First-time setup

```bash
cp ../.env.example ../.env
# fill in DATABASE_URL
cd ..
npx prisma migrate dev --name init
npm run seed
```

`migrate dev` will:
1. Snapshot the current schema state into a SQL migration under `prisma/migrations/<timestamp>_init/`.
2. Run that migration against your `DATABASE_URL`.
3. Regenerate the Prisma client.

## Going to production

```bash
# On the deploy host:
npx prisma migrate deploy
```

`migrate deploy` applies pending migrations only. It never asks questions, never resets, and never runs interactive prompts. The CI workflow runs `migrate status` before deploy and refuses to proceed on drift.

## Adding a new model / column

```bash
# 1. Edit prisma/schema.prisma
# 2. Generate the migration locally
npx prisma migrate dev --name add_something
# 3. Commit the generated migration along with the schema change
```

**Never edit a migration after it has shipped.** If you need to amend something, write a follow-up migration. The CI checksum on `migration_lock.toml` will catch silent edits.

## Multi-tenancy migration plan

The schema ships with nullable `tenantId` columns on every top-level entity. To go multi-tenant:

```sql
-- 1. Backfill — one Tenant row per workspace, every existing record assigned to it.
INSERT INTO "Tenant"(id, slug, name, "createdAt", "updatedAt") VALUES ('default', 'default', 'MyTaskKing', NOW(), NOW());
UPDATE "User"        SET "tenantId" = 'default' WHERE "tenantId" IS NULL;
UPDATE "Channel"     SET "tenantId" = 'default' WHERE "tenantId" IS NULL;
UPDATE "Task"        SET "tenantId" = 'default' WHERE "tenantId" IS NULL;
UPDATE "Lead"        SET "tenantId" = 'default' WHERE "tenantId" IS NULL;
UPDATE "FileAsset"   SET "tenantId" = 'default' WHERE "tenantId" IS NULL;
UPDATE "MeetingRoom" SET "tenantId" = 'default' WHERE "tenantId" IS NULL;
```

Then write a follow-up Prisma migration that flips those columns to `String NOT NULL` and tighten the indexes if you want true tenant isolation at the index level. Finally toggle `MULTI_TENANT=true`.

## Destructive migrations — the rule

Anything that drops a column, drops an enum value, or shrinks a type is a one-way operation in Postgres. The rollback workflow refuses to run them — you must forward-fix with another migration. Plan for that:

- **Drop a column?** Ship the code that stops reading it first. Wait one deploy cycle. Then drop.
- **Rename a column?** Add the new one, dual-write, switch reads, drop the old.
- **Shrink a varchar?** Don't. Add a validation in the service layer instead.

## Read replicas

The repository wrapper ([../src/database/repository.js](../src/database/repository.js)) keeps a second `PrismaClient` pointed at `READ_REPLICA_URL`. Migrations only ever run against the primary — never set `DATABASE_URL` to a replica.

## Local resets

```bash
npx prisma migrate reset    # destroys local DB, replays migrations, runs seed
```

Never run this against staging or production. The script is gated behind a confirmation prompt by default.
